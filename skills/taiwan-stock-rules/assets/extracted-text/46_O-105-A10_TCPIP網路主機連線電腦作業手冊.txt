概說
前言
為方便證券商能以目前世界上最普遍使用的TCP/IP網路與臺灣證券交易所進行各項業務，臺灣證券交易所制定以TCP/IP網路為基礎的主機連線作業規範，並延續原以X.25網路為基礎的「臺灣證券交易所主機連線作業手冊」的精神與原則，繼續提供標準、開放及安全的連線作業環境。

TCP/IP網路主機連線作業方式除針對TCP/IP網路的特性訂定相關的參數及訊息傳送方式外，其他與交易、結算、特殊交易及各項申報作業等的訊息流程及訊息內容都與X.25網路的規定相同。證券商由X.25網路轉換到TCP/IP網路時，只需更改與網路相關的接收及傳送介面程式，其他業務相關的應用程式都可以繼續使用。轉換到TCP/IP網路對證券商現有電腦系統及現行作業流程的影響已經被控制到最小的範圍。

本作業手冊的範圍
本作業手冊僅規範TCP/IP網路的相關設定、訊息傳遞方式及訊息格式，並不涉及交易或結算等業務內容。

有關各項交易、結算、特殊交易或各項申報作業的業務內容及方式請參閱「臺灣證券交易所主機連線作業手冊」（簡稱TMP，TSEC Message Protocol）的第三章到第八章或各項業務的作業手冊。

有關證券商申請數據線路、測試及正式上線等相關規定請參閱「申請競價設備連線及異動作業」。

本作業手冊的訂定與修改
本系統規範由台灣證券交易所電腦規劃部研訂，實施後得視實際需要由台灣證券交易所修訂公佈。

網際網路文件下載
本作業手冊及其他所有業務的作業手冊皆可於臺灣證券交易所官方網站www.tse.com.tw下載。請於進入網站首頁後，選擇「文件下載」，再選擇「電腦規劃部/電腦作業部作業手冊查詢一覽表」。

TCP/IP證券交易資訊網路
臺灣證券交易所TCP/IP證券交易資訊網路（簡稱IP交易網路）架構
臺灣證券交易所IP交易網路採用中華電信數據通訊分公司以ATM架構為骨幹所建置的VPN（Virtual Private Network），證券商與證交所之間連線都設定成點對點固定IP位址的VC（Virtual Circuit），具備優越效能與高度安全的特性。

證券商端的防火牆

證券商端的IP位址分配
證券商端的IP位址由臺灣證券交易所管理及發放。證券商申請使用的每一條數據線路都可分配到一個交易用的IP位址及一個測試用IP位址，不同的數據線路使用不同IP位址，但專為備援用的ADSL線路使用原線路所分配到的IP位址，不另外分配IP位址。

一家證券商可視需要申請多個IP位址（多條線），也可多家總分公司共用一個IP位址（一條線）。

證交所端的設備
如圖2.1.-1所示，證交所端的設備包含主機房、備援機房及測試機房的三個主機。主機房有二個路由器互為備援，其中一個路由器故障時，另一個路由器仍可與證券商連線。備援機房也有二個路由器互為備援，測試機房則有一個路由器。證券商的每一條數據線路都會設定五個PVC到證交所端的每一個路由器上，但ADSL線路則只能建單一PVC連到一個證交所端的路由器。證券商若希望使用ADSL線路連到多個證交所端的路由器，則必須申請多條ADSL線路。

正常情形下，連到備援機房的VC並不啟用，證券商只連線到主機房進行交易。當主機房發生問題無法進行交易時，證交所將關閉證券商連到主機房的VC，並啟用證券商連到備援機房的VC。主機房與備援機房主機採用同樣的IP位址，因此，當證交所從主機房切換到備援機房時，證券商的連線也自動由主機房切換到備援機房，證券商不必作任何線路或IP位址的調整就可以與證交所備援機房連線交易，惟證券商仍需依「集中市場當日全量備援電腦作業手冊」規定配合進行各項查詢檢核作業。

證交所的測試機房可提供證券商上線前測試或假日測試使用，採用與主機房不同的IP位址，以避免發生錯誤。證券商進行測試時，也必須使用測試用IP位址。

證券商連線架構及備援方式
證券商可以依業務的需要及成本的考量，規劃證券商端的備援方式，以預防線路或主機發生問題時無法進行交易的情形。一般而言，除了總分公司的電腦系統可互為備援外，證券商尚可採取ADSL線路備援、多數據線路備援或異地備援等幾種備援方式。

ADSL備援
證券商線路發生問題時，可採用ADSL線路備援。每一路ADSL僅可設定一個VC連到證交所的某一個路由器。如圖2.7.-1所示，證券商若要同時使用ADSL連線到證交所主機房及備援機房，則必須申請二路ADSL。

Socket使用方式
Socket的使用方式類似X.25網路主機連線的X.25 PVC （Permanent Virtual Circuit）使用方式，每一個Socket只能進行一項功能，不同的功能分配在不同的Socket上進行。證券商的任一數據線路（或任一IP位址）上要建立的Socket數量及進行的功能項目，都必須在申請使用時設定。每一數據線路（或每一個IP位址）上的Socket所使用的Source Port Number則由證券商申請時自行設定，不同的Socket必須設定不同的Source Port Number。原則上，Source Port Number的數值區間必須依照功能類別來劃分。

Socket Session的建立由證券商主動發起，證券商為Client端，證交所為Server（或稱Listener）。證交所將以判斷Source IP加上Source Port Number作為辨別Socket的方法。每一個Source IP加上Source Port Number的組合只能建立一個Socket Session。證券商以相同的Source IP加上相同的Source Port Number來建Socket Session時，將被視為重建Socket Session，原已建立的Socket Session會被中止，在該Socket Session上進行的功能也會被中斷。

Socket ID、密碼及分配的功能
每一個Socket也訂定一個2-byte的Socket ID，用來區別Socket，類似X.25主機連線作業中的X.25 PVC ID。每一個Socket也必須設定密碼。Socket建立後，執行連線子系統的登錄作業時必須以Socket密碼進行運算，再將運算值傳給證交所。Socket配置查詢及密碼異動與原X.25 PVC配置查詢及密碼異動方式相同。

每一個Socket都有自己的應用程式功能類別（AP-CODE）。證交所將依據證券商申請競價設備連線時所設定的AP-CODE，與證券商執行登入作業時必須傳送AP-CODE比對，證券商必須傳送正確的AP-CODE才可進行功能。

Socket的備援
證券商若申請使用多數據線路，就可以利用數據線路互相備援。證券商可以預先填寫Socket/Port備援申請表（參閱「申請競價設備連線及異動作業」附件十一），指定某一線路（即某一個IP位址）上未設定執行任何業務的Source Port Number備援另一線路（另一個IP位址）上的某一個Source Port Number（Socket）。一旦有線路發生問題，證券商可以電話聯絡證交所機房進行轉換，以預設的備援用Source IP加上Source Port Number建的Socket取代故障線路上的Socket。Socket備援的作法類似X.25網路主機連線的X.25 PVC備援。
Socket通訊協定
Socket通訊處理層與業務應用程式及TCP/IP網路的關係

圖 4.1 –1 Socket通訊處理層與業務應用程式關係圖

	為了使證券商端的原來的各項功能之應用程式如委託下單、成交回報、檔案傳輸等（簡稱AP），不經過修改就可以連上TCP/IP網路，證券商必須在各項AP的前端加上具有處理Socket能力的介面程式（或稱Socket通訊處理層，簡稱Socket層），來負責Socket的建立與重建，以及各種資料的傳送與接收。

	如圖4.1 –1所示，Socket層介於AP與TCP/IP網路之間，負責將來自AP的TMP資料加上頭尾的識別欄位，變成Socket通訊處理層訊息（簡稱Socket訊息或SLM，Socket Level Message），再透過TCP/IP網路的Socket將SLM傳送到對方。接收方的Socket層則必須將接收到的SLM去除頭尾的識別欄位，再將TMP資料交給AP。

Socket通訊處理層的功能
建立Socket Session
根據各個Socket分配到的Source IP及Source Port Number連接(connect)到證交所的Destination IP及Destination Port Number，建立Socket Session。
包裝及解包裝TMP訊息
因TCP/IP網路上傳輸的資料為byte stream型式，沒有類似X.25網路資料封包的概念，為區別出一筆一筆的訊息，所有TMP的訊息都要加上啟始欄位及結束欄位，形成Socket訊息，請參考「4.5. Socket通訊處理層訊息格式」。
確定Socket Session正常
Socket Session建立後，若證券商或證交所任一方都一直沒有接收到對方傳來的資料時，很難判斷是因為當時並沒有任何AP需要傳送資料，或是有AP需要傳送資料卻因為網路元件發生故障而無法傳送，因此，在一段預設的時間內，若AP沒有任何訊息要傳送，Socket層必須主動傳送確定Socket Session連線訊息（簡稱Heartbeat），使Heartbeat的接收方可以確定Socket Session正常。
重建Socket Session
因網路元件或雙方的電腦軟硬體設備發生故障，造成Socket Session中斷時，Socket層必須判斷發生錯誤的原因，自動排除錯誤後重新建立Socket Session。
結束Socket Session
每日結束所有業務及作業後，結束Socket Session。

Socket通訊處理層作業流程
Socket通訊處理層的作業有：
建立Socket Session作業
TMP訊息傳送作業
確定Socket連線作業
TMP訊息接收作業
結束Socket Session作業


證券商的電腦系統啟動後，應該先執行建立Socket Session作業，才可以進行TMP訊息傳送作業、確定Socket連線作業或TMP訊息接收作業。證券商進行任何一項作業發生問題時，都必須先執行結束Socket Session作業，再重新執行建立Socket Session作業。證券商AP應於每次Socket Session建立之後，執行TMP連線子系統的開機通知作業，登錄作業及應用子系統啟動作業，才可進行各項應用子系統的作業。

證券商於所有業務結束後，執行結束Socket Session作業，再停止電腦系統。

一般流程


說明

與原X.25網路主機連線作業之差異
原「臺灣證券交易所主機連線作業手冊」第三章到第八章之內容全部適用於TCP/IP主機連線作業。但原手冊中有關X.25 PVC的訊息欄位，在TCP/IP主機連線作業中應視為Socket的欄位，例如PVC ID改為Socket ID、PVC密碼改為Socket密碼。TMP中與Socket相關的作業有：
要求傳送連線PVC/Socket配置檔作業（B36）
更改PVC/Socket密碼作業（B37）
普通股委託輸入作業（或其他交易子系統如零股、標借、拍賣）
跨PVC/Socket連線查詢作業

各作業的訊息格式請參考原手冊中之相關規定。

為利用TCP/IP網路在傳輸效能上的優勢，檔案傳輸每筆資料長度在TCP/IP網路上將可以放大到4096 bytes。另外，為了改善原TMP的檔案傳輸功能無法確定雙方連線是否正常的問題，在TCP/IP網路的主機連線作業上也增加檔案傳輸確定連線的作業。證券商可以在連線子系統進行連線時選擇使用原來的檔案傳輸的功能或擴充版檔案傳輸功能，一旦選擇新的檔案傳輸功能，則必須同時配合4096 byte傳輸長度與確定連線作業之相關規定，無法只選擇其中某一項新功能。

證券商選擇使用原檔案傳輸或擴充版檔案傳輸的方法是在連線子系統的登錄訊息L040中，以AP-CODE通知證交所。若AP-CODE為1，表示證券商決定以原檔案傳輸子系統的規格連線，若AP-CODE為9，則表示證券商將使用擴充版的檔案傳輸子系統規格。

證券商即使選擇使用擴充版的檔案傳輸子系統，但在要求傳送連線PVC/Socket配置檔作業（B36）時，證交所回覆的連線PVC/Socket配置檔B36中，檔案傳輸Socket的AP-CODE欄位仍是1。

證券商在連線子系統的登錄作業中選擇使用擴充版檔案傳輸後，檔案傳輸的啟始作業仍使用原來的訊息格式（傳送功能使用F010，接收功能使用F090）。但檔案傳輸中繼作業的傳送資料訊息則必須使用擴充版格式，傳送功能方面使用擴充版格式F210（原格式為F030），傳送資料回覆訊息使用擴充版格式F220（原格式為F040）。而在接收功能方面，使用擴充版格式F230（原格式為F110），傳送資料回覆訊息使用擴充版格式F240（原格式為F120）。各訊息詳細格式內容請參考5.4訊息格式說明。
