概說

1.1前言
為加強整體證券交易資料處理機能，臺灣證券交易所（以下簡稱證交所）規劃電腦主機連線作業規範，使各種廠牌的電腦主機更容易與證交所的電腦主機連線，同時各證券商也可利用此連線方式開發各種應用軟體以滿足內部管理及控制之需求。

在我們規劃主機連線規範時，考慮下列精神與原則︰
 1.標準化－主機連線盡量採用標準化的產品，以減少開發的阻力以及開發所需之人力。例如在連線架構上採用TCP/IP 的通訊協定。
 2.安全性－連線時證券商須輸入密碼，確認無誤後方可連線。電腦主機間的訊息傳輸均透過ＰＶＣ( PERMANENT VIRTUAL CIRCUIT)進行，安全性較高。
 3.擴充性－預留未來系統擴充之彈性。
 4.維護性－主機連線之應用系統較為複雜，設計時必須考慮將來維護工作之簡化。

1.2作業規範訂定目的
由於主機連線作業方式涉及電腦軟體與硬體之規格，以及人工作業之密切配合等複雜因素，為使證券商與交易所之業務能順利的透過電腦主機連線方式進行，特訂定本規範作為證券商電腦主機連線施行之依據。

1.3作業規範適用範圍
本規範訂定之主機連線作業含蓋了所有現行電腦端末作業進行之業務，這些端末作業包括委託輸入、成交回報、結算及投資人管理，這些端末作業均有詳細的操作手冊供證券商參考。本規範同時訂定了證交所與證券商電腦主機間之連線方式及雙方之通訊協定。本規範並未包括主機連線之測試及上線程序，這兩部份請參閱”申請競價設備連線及異動作業”。

1.4作業規範之訂定與修改
本系統規範由台灣證券交易所電腦室研訂，實施後得視實際需要由證交所修訂公佈。
二、 連線架構

2.1主機連線之硬體架構
主機連線之硬體架構其特性如下︰
 1.證券商以「TCP/IP證券交易資訊網路」與證交所主機連線。
 2.所有現行作業(交易、結算及投資人管理)均可利用此一架構進行。

2.2證券商之競價設備與證交所電腦連線方式
證券商之競價設備透過該公司之主機經由證券交易網路與證交所連線。

2.6主機連線之故障處理
主機連線系統因下列原因︰
 1.證交所電腦系統故障(電腦主機或應用系統）。
 2.證券商電腦系統故障(電腦主機或應用系統）。
 3.線路故障(網路、防火牆..）。
 4.訊息回應太慢(TIME OUT）。
致無法將訊息檔案，正確送達對方或無法收到對方送來之訊息檔案，證券商對故障及錯誤的處理，應訂定處理程序，並以系統安全考慮為前提，避免引起更大的錯誤，處理方法說明如下︰
(1)故障排除後，再重新開始連線作業。
(2)可經由分公司幫忙代打輸入委託。
(3)回應訊息太慢︰將相關資料記錄於記錄檔中，請記錄資料送出之時間及回覆時間，並以電話連絡交易所。

2.7 不明訊息及資料錯誤處理說明
為確保訊息資料正確性，每次從證交所接到的訊息資料，都須檢查各欄位資料內容是否正確；檢查範圍可分為下列三部分：
(1)先檢查CONTROL-HEADER及FILE-TRANSFER-HEADER如有錯誤，則歸類
  為不明訊息。
(2)再檢查STATUS-CODE如不為"00"，即必須作錯誤處理。
(3)最後檢查其它資料欄位如有錯誤，即歸類為資料錯誤。
以下就不明訊息處理，STATUS-CODE，資料錯誤處理分別說明。
2.7.1不明訊息處理
 1.不明訊息定義︰對CONTROL-HEADER及FILE-TRANSFER-HEADER
                 收到非預期之內容，即視為不明訊息。
 2.發生時機︰
(1)CONTROL-HEADER無效
檢查CONTROL-HEADER內容不屬於規範說明之範圍，或不屬於原作業子系統之作業。
(2)可能發生之作業
a.連線作業。
b.單筆訊息及檔案傳輸作業。
c.交易作業。
(3)執行狀態不符 即訊息之傳送，接收非按規定之次序出現；如︰作業中應接到回應訊息，卻接到需求訊息。
 3.處理說明︰
(1)記錄錯誤原因。
(2)原作業不作進一步處理，回到連線子系統。
(3)將錯誤原因傳回連線子系統。
(4)重新開始連線作業，並將錯誤原因說明於CONTROL-HEADER之STATUS-CODE中。

2.7.2 STATUS-CODE說明
證交所傳來訊息中，連線作業的開機通知及各項作業的回覆訊息，其STATUS-CODE不為"00"，意即證交所通知證券商前次作業錯誤原因請參閱各子系統錯誤訊息說明。

2.7.3資料錯誤處理
 透過單筆訊息及檔案傳輸之各項作業，資料錯誤不在 STATUS-CODE中說明，
 是因為其內容皆在後序作業處裡。
 1.資料錯誤定義︰ 訊息檔案可明確判知所屬子系統，但有錯誤之資料或狀態。
 2.發生時機︰
(1)證交所透過檔案傳輸接收(pvc)傳送錯誤訊息檔
(2)透過單筆訊息及檔案傳輸之各項作業，如在BODY中資料欄位內容不對
(3)可能發生之作業
a.成交回報補送要求。
b.結算系統作業。
c.投資人管理系統作業。
d.交易報表傳送作業。
(4)作業要求或作業錯誤超過規定次數 如︰相同作業要求超過十五次。
 3.處理說明︰（詳細說明請參閱各子系統錯誤訊息處理說明）。證交所通知證券商錯誤訊息時，表示前次傳送應用系統之資料有錯誤，證券商經更正處理後，仍繼續原先子系統之作業。
(1)接收證交所之錯誤訊息檔
(2)記錄錯誤原因
(3)各子系統一般處理說明
A.成交回報補送作業︰ 依證交所傳送之補送要求錯誤訊息，更正後，再進行補送作業。
B.結算子系統及投資人管理子系統︰
(a)處理證交所傳送之錯誤訊息檔案︰ 依據錯誤檔案之內容，逐筆分析錯誤原因，更改後, 重新傳送。
(b)檢查證交所傳送之資料檔案有誤時︰ 將錯誤資料記錄於錯誤檔案中，通知證交所。
2.8主機連線之備援
證券商可考慮下列各種備援方式︰

1.電腦主機備援︰增設一台電腦主機備援。

2.備援中心︰證券商可以總、分公司相互為備援中心。

三、連線子系統

3.1說明
連線子系統（LINK SUBSYSTEM)提供應用子系統從離線狀態（OFF LINE）進入連線狀態（ON LINE），及從連線狀態進入離線狀態.在線上執行時各pvc 均須先透過連線子系統檢核成功後，才能進入應用系統。
連線子系統之通訊協定設計採交談式（INTERACTIVE），在線上以半雙工（HALF DUPLEX）方式進行。

作業共同點說明：
1.不明訊息︰非連線子系統之訊息，即SUBSYSTEM-NAME不等於10之訊息或訊息之CONTROL HRADER部份任何欄位錯誤皆視為不明訊息，在開機通知作業與登錄作業若接收到不明訊息則由開機通知作業重新開始，若在離線作業接收到不明訊息則強制進入離線狀態。
2.TIMEOUT︰在連線子系統階段，不論接收或傳送訊息，TIMEOUT之時間皆定為三分鐘，以接收或傳送訊息後開始計算，在開機通知作業與登錄作業若發生TIMEOUT則由開機通知作業重新開始，若在離線作業發生TIMEOUT則強制進入離線狀態。

訊息連動關係說明︰
連線子系統使證交所與證券商從離線狀態進入連線狀態需完成三個作業（開機通知作業，登錄作業及應用子系統啟動作業），共包含三個2-WAY的訊息交換，此六個訊息彼此有連動關係，即第一個2-WAY的訊息交換完成後才能進入第二個2-WAY的訊息交換，待其完成後才能進入第三個2-WAY的訊息交換，然後雙方進入各應用子系統執行各項業務，但若第二個2-WAY的訊息交換或第三個2-WAY的訊息交換無法完成，皆需回到第一個2-WAY的訊息交換重新開始。

5.若已完成開機通知作業後仍接收到開機通知訊息，必需回到開機通知作業，請檢查STATUS-CODE錯誤原因後重新啟動。
6.在開機通知作業階段，若對方所回之訊息錯誤，即對方回不明之訊息則主動重送開機通知訊息。

請參考3.6 MESSAGE-ID︰L010。
3.證券商應處理事項
當證券商收到開機通知訊息後則回確認開機通知訊息。

(C)登錄訊息
1.訊息發生情況
收到來自證交所的通知登錄訊息後。

四、單筆訊息與檔案傳輸通訊協定

4.1說明
為滿足證交所與證券商主機連線之檔案傳輸作業需求，單筆訊息與檔案傳輸子系統最主要之工作在證交所與證券商主機間傳送各應用子系統所需之單筆訊息(single message)與檔案(file)，類似一般公司裡收發室之工作，只負責傳遞及收取單筆訊息及檔案，至於單筆訊息及檔案之內容處理，則由各應用子系統負責，而在本子系統中有四種作業︰檔案傳輸起始作業，檔案傳輸中繼作業，單筆訊息傳輸作業，與結束傳輸作業。
4.2單筆訊息與檔案傳輸子系統範圍
4.2.1業務範圍
1.單筆訊息與檔案傳輸子系統上有四種基本作業︰
(1)檔案傳輸起始作業
負責傳送或接收各種資料檔及報表檔之起始訊息。
(2)檔案傳輸中繼作業
負責傳送或接收各種資料檔及報表檔之資料訊息。
(3)單筆訊息傳輸作業
負責傳送或接收各應用系統所需之單筆訊息。
(4)結束傳輸作業
負責結束傳輸系統之操作，回到連線子系統。
4.2.2架構範圍
1.單筆訊息與檔案傳輸子系統所適用之應用子系統如下︰
(1)成交回報補送
(2)交易報表子系統
(3)投資人管理子系統
(4)結算子系統
(5)單筆訊息
2.如圖4.2.2-1所示，單筆訊息與檔案傳輸子系統必須以連線系統建立證交所與證券商主機間之連線及離線狀態，而單筆訊息與檔案傳輸子系統對外透過TCP/IP之通訊協定接收與傳送各應用子系統所需之單筆訊息與檔案。

4.4單筆訊息與檔案傳輸子系統功能
1.單筆訊息與檔案傳輸子系統上有四種基本作業︰
(1)檔案傳輸起始作業
(2)檔案傳輸中繼作業
(3)單筆訊息傳輸作業
(4)結束傳輸作業
2.單筆訊息與檔案傳輸子系統有兩個傳輸介面(傳送與接收介面)，而每個介面都有四種作業，故兩個介面共有八種作業，以下分別介紹︰
(1)檔案傳輸起始作業-傳送功能
(2)檔案傳輸中繼作業-傳送功能
(3)單筆訊息傳輸作業-傳送功能
(4)結束傳輸作業-傳送功能
(5)檔案傳輸起始作業-接收功能
(6)檔案傳輸中繼作業-接收功能
(7)單筆訊息傳輸作業-接收功能
(8)結束傳輸作業-接收功能
3.傳輸協定說明
單筆訊息與檔案傳輸子系統之傳輸協定有以下特色:
(1)證券商與證交所都有兩個傳輸介面(傳送與接收介面)，傳送介面之一方為主動者，接收介面之一方為被動者，分別走不同之virtual circuit。
(2)傳輸協定係採用半雙工(HALF-DUPLEX)之傳輸協定，主要著眼點在容易開發，測試及維護。
(3)本傳輸協定可傳遞兩種標的物，一是檔案，二是單筆訊息，可符合各應用子系統之作業需求。
(4)本傳輸協定具備錯誤處理能力，可依據錯誤之嚴重情形，決定由本身處理或要求連線系統處理。
(5)本傳輸協定可隨時視作業需要中斷正在傳送中之檔案傳輸。

4.由圖4.2.2-2可知，證券商與證交所的傳送介面是主動者(initiator)，而接收介面為被動者(responser)，在以下的介紹中，我們先了解正常狀況之作業情形，而後再根據每種作業之正常及異常狀況做詳細之分析。
(1)檔案傳輸起始作業
檔案傳輸起始作業負責檔案傳輸之起始交談作業，其中包括傳送介面送出起始訊息給接收介面，以及接收介面送出回覆起始訊息給傳送介面，若回覆正確，則進入檔案傳輸中繼作業，否則以異常處理程序處理。


(2)檔案傳輸中繼作業
檔案傳輸中繼作業負責傳輸檔案之資料內容，其中包括傳送介面送出傳送資料訊息給接收介面，以及接收介面送出回覆資料訊息給傳送介面，若回覆正確，則繼續傳送下一筆資料訊息直到EOF(End of File)為止，若回覆錯誤，則以異常處理程序處理，當處理完最後一筆資料訊息且回覆正確時，則傳送介面回到傳輸子系統之開頭，處理下一個作業。



(3)單筆訊息傳輸作業
單筆訊息傳輸作業負責傳送單筆訊息，所謂單筆訊息是指僅傳輸一筆訊息，不像傳輸一個檔案要傳輸多筆訊息(含起始及資料訊息)，至於單筆訊息之意義是由應用系統去解釋，此作業包括傳送介面送出傳送單筆訊息給接收介面，以及接收介面送出回覆單筆訊息給傳送介面，若回覆正確，則傳送介面回到傳輸子系統之開頭，處理下一個作業，否則以異常處理程序處理。



(4)結束傳輸作業
結束傳輸作業負責結束傳輸子系統之操作，回到連線子系統，其中包括傳送介面送出結束訊息給接收介面，以及接收介面送出回覆結束訊息給傳送介面，若回覆正確，則傳送介面回到連線子系統，否則以異常處理程序處理。



5.共同處理事項 在以下之討論中，我們以證券商應處理事項為討論主體。
(1)不明訊息︰ 指訊息欄位中HEADER值(請參考4.5代碼系統)不正確之訊息。對證券商傳送及接收介面而言，若收到不明訊息時，應不做任何回覆後，直接回到連線子系統，要求重新連線，以及不正確情形對應之status code，表示雙方已在不同步狀態，必須重新建立連線關係。
(2)TIMEOUT︰ 對傳送介面而言，當送出一筆訊息後，必等待回覆訊息，此時設定TIMEOUT時間為60秒，當對方沒有回覆時，傳送方回到連線子系統，要求重新連線，表示雙方已在不同步狀態，必須重新建立連線關係。 對接收介面而言，當連線完成後，則作等待接收訊息之動作。要一直等到對方送來之起始訊息，或傳送單筆訊息，或結束訊息，但進入檔案傳輸中繼作業時，此時發出之接收指令必須設定TIMEOUT為60秒，因為傳送檔案為多筆訊息之連動作業，必須等到下一筆資料訊息之到來。
(3)錯誤處理︰ 對傳輸子系統而言，錯誤處理程序可分為嚴重錯誤及輕微錯誤之處理程序。所謂嚴重錯誤是指接收到不明訊息，此時應不做任何回覆，直接回到連線子系統，要求重新連線，以及不正確情形對應之status code('8'或'9'開頭者)，表示雙方已在不同步狀態，必須重新建立連線關係. 所謂輕微錯誤是指接收到之訊息為status code不為'8'開頭或'9'或'00'之發生情形，此時傳輸子系統結束該訊息所對應之作業，然後回到傳輸子系統之開頭，處裡下一個作業。

b. 若為 illegal function code ，則 status code = '82'

i. 若為 internal error        ，則 status code = '89'
j. 若為 illegal file code     ，則 status code = '10'

證券商收到證交所傳來之傳送單筆訊息，經檢查正確後，送回正確回覆單筆訊息給證交所。
2.訊息格式與內容︰
請參考4.5 Message-ID︰F140。
3.證券商應處理事項︰
(1)證券商收到證交所傳來之傳送單筆訊息，必須檢查訊息中各欄位之正確性以及是否符合各應用子系統之作業需求。
(2)當接收到正確訊息，證券商將此訊息做完內部處理，然後送出正確回覆後，即算是處理完此單筆訊息，則證券商回到傳輸子系統之開頭，等待下一筆訊息之到來。
(C)錯誤回覆單筆訊息(Negative Confirmation for Send Single Message)︰
1.訊息發生情況︰
證券商收到證交所傳來之傳送單筆訊息，經檢查有錯誤時，送
回錯誤回覆單筆訊息給證交所.


2.訊息格式與內容︰
請參考4.5 Message-ID︰F140。
3.證券商應處理事項︰
(1)當收到傳送單筆訊息，經檢查後發現有錯誤時，status code如下︰(各status code之詳細意義見4.5節)
a. 若為 illegal subsystem name，則 status code = '81'
b. 若為 illegal function code ，則 status code = '82'
c. 若為 illegal message type  ，則 status code = '83'
d. 若為 illegal message time  ，則 status code = '84'
e. 若為 illegal status code   ，則 status code = '85'
f. 若為 illegal source id     ，則 status code = '86'
g. 若為 illegal object id     ，則 status code = '87'
h. 若為 illegal body length   ，則 status code = '88'
i. 若為 internal error        ，則 status code = '89'
l. 若為 illegal file code     ，則 status code = '10'
m. 若為 illegal EOF value     ，則 status code = '11'
n. 若為 illegal file size     ，則 status code = '12'
r. 若為 abort by initatior    ，則 status code = '19'
(2)則處理步驟如下︰
同4.4.5.2(C)錯誤回覆起始訊息之3.證券商應處理事項之(2)處理步驟

4.4.8結束傳輸作業-接收功能
4.4.8.1一般流程與說明


1.說明
2-WAY的訊息交換。
2.目的
證交所欲結束單筆訊息與檔案傳輸作業。
3.發生條件
當證交所完成單筆訊息與檔案傳輸作業後，必須傳送結束訊息給證券商，以通知單筆訊息與檔案傳輸作業之完成。
4.完成後影響
證券商將此訊息做完內部處理，然後送出回覆後，證券商回到連線子系統，傳回要求結束參數，進行離線作業。

4.4.8.2訊息流程與訊息格式


 (A)結束訊息(End Message)︰
1.訊息發生情況︰
當證交所完成單筆訊息與檔案傳輸作業後，必須傳送結束訊息給證券商，以通知單筆訊息與檔案傳輸作業之完成。
2.訊息格式與內容︰
請參考4.5 Message-ID︰F150。
3.證券商應處理事項︰
證券商連線完成或處理一完整交易後，要發出接收指令，一直等到交易所送來之結束訊息(作業八)或其他訊息(其他作業)，此時不必設定TIMEOUT。
(B)正確回覆結束訊息(Positive Confirmation for End Message)︰
1.訊息發生情況︰
證券商收到證交所傳來之結束訊息，經檢查正確後，送回正確回覆結束訊息給證交所。
2.訊息格式與內容︰
請參考4.5 Message-ID︰F160。
3.證券商應處理事項︰
(1)證券商收到證交所傳來之結束訊息，必須檢查訊息中各欄位之正確性以及是否符合各應用子系統之作業需求。
(2)證券商將此訊息做完內部處理，然後送出正確回覆後，結束單筆訊息與檔案傳輸作業，證券商回到連線子系統，傳回要求結束參數，進行離線作業。
(C)錯誤回覆結束訊息(Negative Confirmation for End Message)︰
1.訊息發生情況︰

證券商收到證交所傳來之結束訊息，經檢查有錯誤時，送回錯誤回覆結束訊息給證交所。
2.訊息格式與內容︰
請參考4.5 Message-ID︰F160。
3.證券商應處理事項︰
(1)當收到結束訊息，經檢查後發現有錯誤時，status code如下︰(各status code之詳細意義見4.5節)
a. 若為 illegal subsystem name，則 status code = '81'
b. 若為 illegal function code ，則 status code = '82'
c. 若為 illegal message type  ，則 status code = '83'
d. 若為 illegal message time  ，則 status code = '84'
e. 若為 illegal status code   ，則 status code = '85'
f. 若為 illegal source id     ，則 status code = '86'
g. 若為 illegal object id     ，則 status code = '87'
h. 若為 illegal body length   ，則 status code = '88'
i. 若為 internal error        ，則 status code = '89'
k. 若為 illegal file code     ，則 status code = '10'
l. 若為 abort by initatior    ，則 status code = '19'
(2)而不論任何情形，證券商都做完內部處理後，回到連線子系統，傳回要求結束參數，進行離線作業，表示傳輸作業之完成. (此時若status code不為'8'或'9'者，仍必須送出錯誤回覆結束訊息)

 1.Fxx0表示xx為02，04，06，...，16等回覆訊息。
 2.status code為'81'-'88'者(header error)︰
視所收到之訊息為不明訊息，不送出任何回覆後，回到連線子系統，並傳回此status code。
 3.status code為'89'者(internal error)︰
對所收到之訊息，本身主機系統發生無法處理情形時，不送出任何回覆後，回到連線子系統，並傳回此status code。
 4.status code為'92'者(FT count error)︰
當系統收到訊息，累計錯誤次數超過限制時，不送出任何回覆後，回到連線子系統，並傳回此status code。
 5.status code為'93'者(request count error)︰
傳輸某特定檔案之次數超過限制時，不送出任何回覆後，回到連線子系統，並傳回此status code。
 6.status code為'10'者(illegal file code)︰
收到之訊息中之file code欄位值不正確，送出錯誤回覆訊息後，回到傳輸子系統之開頭，處理下一個作業。
 7.status code為'11'者(illegal EOF value)︰
收到之訊息中之EOF欄位值不正確，送出錯誤回覆訊息後，回到傳輸子系統之開頭，處理下一個作業。
 8.status code為'12'者(illegal file size)︰
收到之總資料訊息長度與起始訊息中file size不符合，送出錯誤回覆訊息後，回到傳輸子系統之開頭，處理下一個作業。
       9.status code為'13'者(timing error)︰
收到之訊息出現在規定之作業時間之外，送出錯誤回覆訊息後，回到傳輸子系統之開頭，處理下一個作業。
10.status code為'19'者(abort by initiator)︰
由傳送介面主動中斷檔案傳輸中繼作業時，可將此status code放在訊息中傳出，當收到回覆訊息後，回到傳輸子系統之開頭，處理下一個作業。
11.status code為'20'者(busy now, resend later )︰
在交易時間內，傳輸代號B、C開頭之申報或查詢作業因系統忙碌中，交易所端送出之回覆訊息。當收到回覆訊息後，回到傳輸子系統之開頭，處理下一個作業。

五、委託通訊協定

5.1
為配合應用系統之特性而設計發展。交易系統之特性在於每一筆訊息都必須等待對方之回應。證券商委託輸入之後必須等待證交所送回委託回報，證券商正確接收後才可輸入下一筆委託。

5.2系統功能
 1.所有作業都由證券商主動，證交所為被動
 2.所有作業互相獨立，沒有先後的關係。
 3.委託輸入及回報作業，確定連線作業及重新連線後查詢作業等三項作業走的是同一條 PVC。證券商在連線後或收到上一筆作業的回覆訊息後的一分鐘內，必須發動上述三項作業中的任何一項,否則證交所將視雙方主機為離線狀態，會回到連線子系統，重新連線。
 4.斷線跨PVC查詢作業，即是經由另一功能正常之PVC查詢某指定已斷線PVC寫入檔案中之最後一筆正確委託資料，因此只有當某PVC產生斷線且無法恢復時，才可以執行此作業。
 5.交易所電腦系統之主機連線委託程式，將最後一筆正確之委託資料保存於程式MEMORY中，只有當PVC在斷線或離線時，交易所電腦系統才會將當時PVC之最後一筆正確委託資料寫入檔案中記錄。
 6.所有訊息均由CONTROL-HEADER + BODY 搭配組成。
 7證交所在下列情形下會回到連線子系統重新連線，並帶回上一筆委託之發生狀況於錯誤訊息代號中(STATUS-CODE)，證券商若於委託中收到證交所傳來之連線訊息，務必檢查錯誤訊息代號(不為0 者)，查明原因後，重新連線。

六、成交回報通訊協定

6.1說明
證券商輸入的買賣申報一經成交，證交所電腦主機即將成交資料傳給證券商之電
主機，再由證券商的電腦主機負責列印"成交回報單"。

6.2系統功能
 1.起始作業由證券商主動，證交所為被動，確定連線作業及傳送成交回報作業及結束成交回報等三項作業，證交所均主動傳送。
 2.證券商在完成起始作業連線後，每當超過一分鐘無成交回報資料時，為了確定雙方仍處於連線狀態，必須進行確定連線作業。
 3.所有訊息均由CONTROL-HEADER + AP-MSG + BODY 搭配組成。
 4.證交所在下列情形下會回到連線子系統重新連線，並帶回上一筆委託之發生狀況於錯誤訊息代號中(STATUS-CODE)，證券商若於接收中收到證交所傳來之連線訊息，務必檢查錯誤訊息代號(不為0者)，查明原因後，重新連線。
錯誤訊息代號             訊息內容
91					TIME OUT (未收到證券商傳送之訊息)
92					MSG LENGTH ERR (訊息長度不對)
94            RCV/SND MSG ERR (送/收訊息不成功，線路問題)
95					UNKNOW MESSAGE (收到傳來之不明訊息)

6.3.成交回報作業流程
 1.正常成交回報作業
 2.異常情況處理

證券商配合事項說明如下
 1.正常成交回報作業流程說明：
(1)啟動成交回報起始作業
證券商必須於連線作業完成後之一分鐘內，必須先啟動起始作業。
(2)接收成交回報資料
(A)於開盤後，證券商當日之買賣申報一經成交，證交所即將成交回報資料傳給該證券商。
(B)證券商須按規定格式接收成交資料。
(3)接收確定連線作業： 證券商在完成起始作業連線後，每當超過一分鐘無成交回報資料時，為了確定雙方仍處於連線狀態，必須進行此作業。證交所將主動傳送確定連線訊息，若證交所在一分鐘內未收到證券商送來的確定連線回覆訊息，將視雙方主機處於離線狀態，證交所會回到連線子系統重新建立連線。
(4)結束成交回報資料之傳遞： 交易時間結束，證交所將所有成交回報資料傳送完畢後，會自動傳遞一結束訊息給該證券商，證券商接收到此一結束訊息，核對成交資料總筆數無誤後，應結束成交回報接收子系統，回到連線子系統，結束連線。

 2.異常情況處理程序說明：
(1)成交回報正常作業期間若遇線路中斷，或證券商主機無法運作等情形，證券商應速與證交所連絡，以備援線路維持作業。
(2)證券商如收到不正常的成交回報資料，則處理方法有：
(A)利用檔案傳輸子系統要求補送成交資料。
(B)打電話至證交所電腦室作業組，請求查明錯誤原因。

證交所會回到連線子系統重新建立連線。
